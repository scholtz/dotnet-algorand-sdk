### G1 — `Utils.Utils.GetRandomAssetMetaHash()` uses `System.Random`, not a CSPRNG
- **Severity**: Informational · **Confidence**: Confirmed · **Status**: New (not raised
  in any prior audit cycle — previous cycles' repo-wide RNG sweeps were scoped to the
  week's diff and to `Crypto`/`Account`, not a full-tree grep for `new Random(`)
- `dotnet-algorand-sdk/Utils/Utils.cs:89-96`:
  ```csharp
  public static string GetRandomAssetMetaHash()
  {
      Random rd = new Random();
      byte[] bts = new byte[32];
      rd.NextBytes(bts);
      return Convert.ToBase64String(bts);
  }
  ```
  This is a public static helper (referenced in generated API docs,
  `docs/api/Algorand.Utils.Utils.html`) that produces a 32-byte value intended for the
  `AssetParams.MetadataHash` field used at asset-creation time. It is **not** part of
  the key-generation or signing path — `Account()`'s real key generation
  (`dotnet-algorand-sdk/Algod/Model/Account.cs:81`, `new SecureRandom()`, BouncyCastle's
  OS-CSPRNG-backed generator) is unaffected and was re-confirmed unchanged this cycle.
  `AssetParams.MetadataHash` is a public, on-chain value (typically meant to be a
  content hash of some off-chain asset document) with no confidentiality requirement, so
  this is not a key-leakage risk.
- **Scenario / concern**: The method name and lack of an XML-doc caveat could lead a
  developer to assume the returned value carries some cryptographic guarantee (e.g.
  "unpredictable enough to prevent hash-grinding/collision games around an asset's
  metadata"), when in fact `System.Random` is seeded from a low-entropy, time-based
  default and is trivially predictable to an attacker who can narrow down the
  approximate creation time. Since the field is public metadata rather than a secret,
  the practical impact is low, but it's a correctness/API-honesty gap worth closing.
- **Recommendation**: Either (a) switch to
  `System.Security.Cryptography.RandomNumberGenerator.Fill(bts)` for consistency with
  the rest of the codebase's stated RNG policy (per this audit framework's own item 1
  criterion), or (b) add an XML-doc comment clarifying this produces an arbitrary
  placeholder value with no cryptographic or content-integrity guarantee, so a developer
  who does need one won't rely on this helper's default behavior. Low priority — no
  exploitation path leads to key or fund loss.
