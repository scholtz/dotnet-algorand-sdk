# Security Audit — dotnet-algorand-sdk — 2026-07-16

Audited commit: `97a6c7905b7cd20c7d7ce6bd912fb76d7cdd9ea7`
Auditor: AI agent (Claude) per `audits/AI-AUDITS-INSTRUCTIONS.md`
Previous audit: none — first audit

## Summary

As of this commit, no evidence was found that the SDK transmits, logs, or otherwise
leaks a user's private key or mnemonic. Key generation uses a proper CSPRNG
(BouncyCastle `SecureRandom`), signing is fully local (BouncyCastle `Ed25519Signer`,
no network calls), and the `Account`/`KeyPair` types correctly exclude key material
from JSON/MessagePack serialization and from `ToString()`. The one place private keys
*do* cross the network is the Kmd client's `ImportKeyAsync`/`ExportKeyAsync`/
`ExportMasterKeyAsync` calls — this is inherent to Kmd's documented remote
key-management API and only fires on an explicit, developer-initiated call, not as a
side effect of normal signing/transaction flows.

The most actionable findings are hygiene issues rather than active leaks: private key
bytes are never zeroed/disposed after use, the BouncyCastle crypto dependency is
vendored as an unpinned local DLL rather than the official NuGet package, and
`ApiException` embeds raw HTTP response bodies into exception messages, which is a
narrow but real risk specifically around the Kmd export endpoints (a logged exception
during an export call could carry exported key material into application logs).

## Findings

### F1 — Private key material is never zeroed/disposed from memory
- **Severity**: Low · **Confidence**: Confirmed · **Status**: New
- `dotnet-algorand-sdk/Crypto/KeyPair.cs:51` — `ClearTextPrivateKey` is a plain
  `byte[]` with a public getter and no `IDisposable`/explicit-clear method anywhere on
  `KeyPair` or `Account` (`dotnet-algorand-sdk/Algod/Model/Account.cs`).
- **Scenario**: A long-lived process holds an `Account` for its whole lifetime; the raw
  32-byte private key sits in managed memory (and potentially gets copied by the GC
  during compaction) for as long as the object is reachable, with no way for the
  application to proactively scrub it even after it's done signing.
- **Recommendation**: Add an explicit `Dispose()`/`ClearKey()` path that
  `Array.Clear()`s `ClearTextPrivateKey` (and the BouncyCastle
  `Ed25519PrivateKeyParameters` internal buffer, if reachable) once a developer is done
  with an `Account`. Document that this is best-effort only (the CLR does not guarantee
  no copies remain).

### F2 — BouncyCastle vendored as an unpinned local DLL, not the official NuGet package
- **Severity**: Medium · **Confidence**: Confirmed · **Status**: New
- `dotnet-algorand-sdk/dotnet-algorand-sdk.csproj:130` and `:414-417` reference
  `Libs\BouncyCastle.Crypto.dll` via `<Content>`/`<Reference HintPath=...>` rather than
  a versioned `PackageReference` to the official `BouncyCastle.Cryptography` (or
  legacy `BouncyCastle`) NuGet package. A second copy,
  `ClientGenerator\Libs\BouncyCastle.Crypto.Signed.dll`, exists with a different name,
  suggesting at least two vendored copies may exist in the tree.
- **Scenario**: There is no NuGet-level version pin, no reproducible-build guarantee,
  and no automated path (e.g. Dependabot) to learn about a CVE in BouncyCastle's
  Ed25519 implementation — the ed25519 signing primitive this entire SDK's key
  security rests on. A consumer or auditor cannot verify the DLL's provenance/hash
  against upstream without manually decompiling/hashing it.
- **Recommendation**: Migrate to the official `BouncyCastle.Cryptography` NuGet
  package pinned to a specific version, and remove the vendored binary once parity is
  confirmed (this may already be blocked by a netstandard2.1/Unity compatibility
  reason — if so, document why in the csproj as a comment and record the exact
  upstream BouncyCastle version/commit the vendored DLL corresponds to).

### F3 — `ApiException` embeds raw HTTP response bodies, risking key leakage into logs specifically around Kmd export calls
- **Severity**: Medium (narrow scope) · **Confidence**: Confirmed · **Status**: New
- `dotnet-algorand-sdk/ApiException.cs:17` builds `Message` by concatenating up to the
  first 512 characters of the raw HTTP response body; the full raw body is also kept
  verbatim in the public `Response` property (no size limit).
- `dotnet-algorand-sdk/Kmd/Kmd.cs` — `ExportKeyAsync`/`ExportMasterKeyAsync` responses
  contain a private key / master derivation key in the JSON body returned by Kmd.
- **Scenario**: A developer calls `kmd.ExportKeyAsync(...)`, and something downstream
  of a successful HTTP call throws (e.g. a deserialization mismatch, or the developer's
  own error-handling wraps it) such that an `ApiException` is constructed from that
  response. If the application logs `exception.Message` or `exception.ToString()` at
  error/info level (a very common default in ASP.NET/serilog/etc. exception
  middleware), the exported private key/master key ends up in that log line. This is
  not unique to this SDK's design (any exception-message-includes-response-body
  pattern has this shape) but is worth flagging precisely because two of Kmd's
  legitimate documented operations return raw secret key material in the response
  body.
- **Recommendation**: For the Kmd client specifically, consider not routing
  export-endpoint responses through the generic `ApiException(message, ..., response,
  ...)` path with the body inlined into `Message` — e.g. keep `Response` available for
  programmatic access but redact/omit it from `Message`/`ToString()` for the export
  endpoints, or clearly document in XML doc comments on `ExportKeyAsync`/
  `ExportMasterKeyAsync` that callers must not log exceptions raised by these calls at
  a verbose level.

### F4 — Leftover `Console.WriteLine` in library code (`Signature.cs`)
- **Severity**: Informational · **Confidence**: Confirmed · **Status**: New
- `dotnet-algorand-sdk/Signature.cs:216` — `LogicsigSignature.Verify()` catches a
  signature-verification exception and does `Console.WriteLine("Message = " +
  err.Message)` before returning `false`. Only the exception message text (not key
  material — this path verifies a signature against a already-public address/logic
  sig, not private key data) is printed, so there is no secret-leak risk here, but a
  library unconditionally writing to `Console` is a design smell (surprises consumers
  who don't expect their library dependency to write to stdout, and this behavior
  cannot be turned off).
- **Recommendation**: Remove the `Console.WriteLine`, or replace with a no-op / an
  injectable logger consistent with `Gossip/GossipWebsocketClient.cs`'s use of
  `ILogger<T>`.

### F5 — Kmd `ImportKeyAsync`/Export* calls transmit or receive raw key material by design — flagged for documentation, not as a bug
- **Severity**: Informational · **Confidence**: Confirmed · **Status**: New (documented
  as inherent risk in `RISKS.md` `RISK-004`)
- `dotnet-algorand-sdk/Kmd/Kmd.cs` (`ImportKeyAsync`, `ExportKeyAsync`,
  `ExportMasterKeyAsync`, `SignTransactionAsync`, `SignMultisigTransactionAsync`,
  `SignProgramAsync`) are thin generated wrappers over Kmd's own REST API. Ordinary
  signing flows (`Transaction.Sign(Account)`, `LogicsigSignature.Sign`) never touch
  these endpoints or the network at all — they are fully local. Kmd usage is opt-in
  and only meaningful when a developer runs/points at a Kmd daemon they trust (e.g.
  AlgoKit LocalNet).
- **Recommendation**: No code change needed. Recommend the README/XML docs on the Kmd
  client explicitly state: "Kmd is a remote key-custody service — only point this
  client at a Kmd instance you control and trust; `ImportKeyAsync` sends a raw private
  key over the wire to that instance by design."

## Risk register changes

Initial population of `audits/RISKS.md` with 8 risks (`RISK-001` through `RISK-008`)
covering: private key memory hygiene (ties to F1), BouncyCastle provenance (F2),
Kmd export/log leakage (F3), Kmd's inherent remote-custody model (F5), developer
misuse patterns the SDK cannot control (hardcoding mnemonics, logging exceptions,
copy-pasting insecure sample code), malicious/compromised Algod-Kmd-Indexer node
operators a developer chooses to trust, OS/host-level compromise, and ARC56
client-generator supply chain (Docker image integrity). See `RISKS.md` for full
detail and 5-year misuse-likelihood estimates.

## Recommendations for maintainers

1. (Medium priority) Replace the vendored `BouncyCastle.Crypto.dll` with the official
   pinned NuGet package, or document precisely why that isn't currently possible and
   which upstream version/commit the vendored binary matches (F2).
2. (Medium priority, narrow scope) Redact response bodies from `ApiException.Message`
   for Kmd export endpoints, or add explicit XML-doc warnings on
   `ExportKeyAsync`/`ExportMasterKeyAsync` about exception logging (F3).
3. (Low priority) Add best-effort key-wiping (`Dispose`/`Array.Clear`) support to
   `KeyPair`/`Account` (F1).
4. (Informational) Remove the stray `Console.WriteLine` in `Signature.cs:216` (F4).
5. (Informational, documentation only) Add an explicit security note to the Kmd client
   documentation about its remote-custody trust model (F5).
6. Cover the Unity ILRepack build variant and a live Gossip mainnet check in the next
   audit cycle (not reviewed this time).

## Remediation update — 2026-07-16 (same-day follow-up)

All four actionable findings (F1–F4) were fixed and verified in the working tree
following this audit, on top of the same audited commit
(`97a6c7905b7cd20c7d7ce6bd912fb76d7cdd9ea7`). F5 required no code change (documented as
inherent to Kmd's design). Per-finding outcome:

- **F1 — fixed.** `KeyPair` (`dotnet-algorand-sdk/Crypto/KeyPair.cs`) and `Account`
  (`dotnet-algorand-sdk/Algod/Model/Account.cs`) now implement `IDisposable`.
  `KeyPair.Dispose()` zeroes `ClearTextPrivateKey` via `Array.Clear` and drops the
  BouncyCastle key object references; `Account.Dispose()` delegates to it. Verified by
  new tests `AccountTests.testDisposeWipesPrivateKey` and
  `AccountTests.testKeyPairDisposeIsIdempotent` (both pass).
- **F2 — fixed.** `dotnet-algorand-sdk.csproj` now references the official
  `BouncyCastle.Cryptography` NuGet package (v2.6.2) instead of the vendored
  `Libs\BouncyCastle.Crypto.dll`; the vendored DLL was deleted from the repo. The
  redundant direct `HintPath` references in `test/test.csproj` and
  `sdk-examples/sdk-examples.csproj` were removed (both already get BouncyCastle
  transitively via their existing `ProjectReference` to the main SDK project).
  `ILRepack.Targets` was updated to match the new assembly filename
  (`BouncyCastle.Cryptography` instead of `BouncyCastle.Crypto`) so the Unity build's
  IL-merge step still picks it up. Note: the `ClientGenerator/AlgoStudio.csproj`
  reference to a historical, separately-published `Algorand2` NuGet package's own
  bundled `BouncyCastle.Crypto.dll` was intentionally left untouched — that's an
  unrelated, already-published external artifact, not the local vendored copy this
  finding was about.
  Verified: `dotnet restore` and `dotnet build --configuration Release` both succeed;
  `AccountTests.testSignsTransactionE2E` (a full end-to-end sign against a golden
  reference transaction) still passes; the full `SerialisationTests` msgpack/canonical
  encoding suite (23/23) still passes; and live Arc56 tests against a local AlgoKit
  node show transactions signed with the new crypto package being accepted and
  evaluated by real algod (the only failures observed were algod-side "overspend"
  responses from unfunded LocalNet test accounts, an environment/funding issue
  unrelated to this change — see below).
  **Known limitation**: the Unity/Unity_debug build configuration currently fails to
  compile in this environment for an unrelated, pre-existing reason (missing
  `SerializeField`/`Tooltip`/`InspectorName` types from the vendored Unity stub DLLs) —
  confirmed present before this change too (reproduced identically against the
  pre-audit commit via `git stash`). This means the ILRepack step itself (and thus the
  `BouncyCastle.Cryptography` filename-filter update) could not be exercised
  end-to-end in this environment; the filename filter was updated to match the new
  package's actual assembly name and is a low-risk textual change, but a maintainer
  with a working Unity toolchain should confirm the Unity NuGet package still builds
  and IL-merges correctly.
- **F3 — fixed.** `ApiException.cs`'s constructor and `ToString()` no longer embed the
  raw HTTP response body in `Message`/`ToString()` output; they now report only the
  response length, with a note that the full body remains available via the `Response`
  property for callers who deliberately want it. A comment was added noting this is a
  hand-patch that diverges from the raw NSwag-generated default and should be
  reapplied if the file is ever regenerated from scratch. Verified by new tests
  `ApiExceptionTests.testMessageAndToStringDoNotLeakResponseBody` and
  `ApiExceptionTests.testMessageHandlesNullResponse` (both pass); confirmed no existing
  test asserted on the old `ApiException.Message`/`.ToString()` format.
- **F4 — fixed.** The stray `Console.WriteLine` in
  `Signature.cs` (`LogicsigSignature.Verify`) was removed; the catch block now simply
  returns `false`.

Test evidence: `dotnet test test/test.csproj` — 113/147 passed, with all 34 failures
confined to `Arc4Tests`/`Arc56Tests`/`AVMTypes*` (Arc4/Arc56 contract-deployment tests
that, per `CLAUDE.md`, require a running, funded AlgoKit LocalNet — not present/funded
in this environment; failures were algod-side "overspend" responses, not crypto or
signing errors, and were reproduced as pre-existing by nature of requiring LocalNet
regardless of these changes). `dotnet test SerialisationTests/serialisation-tests.csproj`
— 23/23 passed.
