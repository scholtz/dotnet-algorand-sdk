## Summary

Two threads of work this cycle:

1. **G1/`RISK-011` fixed**: `Utils.GetRandomAssetMetaHash()` now uses
   `RandomNumberGenerator.Fill` instead of `System.Random`. No key-security impact
   either way (confirmed in the original finding); this closes the gap between the
   method's implicit expectations and its actual guarantee.
2. **The Unity build, previously undiagnosed across three prior audit cycles as "blocked
   by a pre-existing compile failure," was actually broken by several independent,
   fixable bugs** — not a fundamental toolchain limitation. All are now fixed, and the
   `Unity`/`Unity_debug` configurations build a real, verified `Algorand4_Unity.nupkg`
   end-to-end in this environment for the first time across all four audit cycles.

None of this cycle's fixes touch the signing/key-material path. `Account`'s real key
generation (`SecureRandom`) is untouched; the Unity fixes are entirely in
model/transaction serialization plumbing, MSBuild build orchestration, and a new
small standalone build-time helper task. No new key-material exposure surface was
introduced.

## What was actually wrong with the Unity build (previously misdiagnosed as one blocker)

Prior cycles all stopped at the first compile error
(`Algod/Model/StateProofTrackingData.cs` missing `SerializeFieldAttribute` etc.) and
recorded it as *the* blocker. Fixing it revealed four more independent problems in
sequence:

1. **Missing `using UnityEngine;`** in three hand-written (non-generated) partial
   classes — `StateProofTrackingData.cs`, `Transactions/AssetCloseTransaction.cs`,
   `Transactions/AssetMovementsTransaction.cs`. The generated sibling files already had
   this; these three hand-written files never did. Fixed by adding the guarded
   `#if UNITY using UnityEngine; #endif` block matching the generated-file convention.
2. **Missing `[IgnoreMember]` under `#if UNITY`** on six properties (`Transaction.cs`'s
   `ConfirmedRound`/`Committed`/`ReceiverRewards`/`SenderRewards`/`CloseRewards`,
   `ApplicationCreateTransaction.ApplicationIndex`, `AssetCreateTransaction.AssetIndex`,
   `AssetTransferTransaction.AssetClosingAmount`, `PaymentTransaction.ClosingAmount`) —
   the `#else` (non-Unity) branch had `[IgnoreMember]` so MessagePack knows to skip
   these local-only properties; the `#if UNITY` branch never did, so the analyzer
   (MsgPack004) correctly flagged them as ambiguous under a `[MessagePackObject]` type.
   Added `[IgnoreMember]` to each Unity branch to match.
3. **A misplaced `#endif`** in `SignedTransaction.Generated.cs`: the `Tx` property's own
   `#if UNITY/#else/#endif` block was nested inside the *previous* property's `#else`
   branch, so under `UNITY` the entire `Tx` property (both variants) silently
   disappeared, breaking every caller (`Transaction.cs`, `JsonConverters.cs`, the class
   itself). Fixed by closing the `Sig` property's conditional before starting `Tx`'s.
4. **An `#if UNITY` branch that was never added** for the `UpdateApplication` case in
   `ClientGenerator/Clients/ProxyBase.cs` (the `CreateApplication` case right above it
   already had one) — assigning a `ulong?`-typed expression to a `ulong`-typed target
   only under Unity, since `StateSchema.NumByteSlice`/`NumUint` become non-nullable
   `ulong` under `UNITY` (by design, since Unity's Inspector doesn't serialize nullable
   value types cleanly). Added the missing branch, mirroring the existing
   `CreateApplication` pattern.

With those four fixed, the Unity **compile** succeeded, but the **ILRepack merge and
packaging steps failed for entirely separate reasons**:

5. **ILRepack.Lib.MSBuild 2.1.18's own `<ILRepack>` task is fundamentally incompatible
   with any current .NET SDK**: its `ResourcesRepackStep.GenerateRepackListResource()`
   unconditionally calls `BinaryFormatter.Serialize`, whose (de)serialization APIs were
   removed outright in .NET 9 (not just gated behind a switch — the call throws
   unconditionally). The underlying `ILRepacking.ILRepack` engine (bundled in the same
   package's `ILRepack.dll`) exposes a `NoRepackRes` option that skips this step, but the
   packaged MSBuild `<ILRepack>` task wrapper never surfaces it, and the package hasn't
   been updated since 2022 (2.1.18 is the latest version on nuget.org; a community fork,
   `ILRepack.Lib.MSBuild.Minor` 2.1.18.2, predates .NET 9 and doesn't fix this either).
   **Fix**: added `BuildTasks/IlRepackMergeTask/`, a small standalone MSBuild task
   project that drives `ILRepacking.ILRepack` directly with `NoRepackRes=true`. It's a
   normal compiled SDK project (not a `RoslynCodeTaskFactory` inline task) because an
   inline task's manually-specified `<Reference>` list replaces the compiler's implicit
   corlib/netstandard references entirely — every attempt to hand-supply a reference
   list for an inline task hit `CS0518: Predefined type 'System.Object' is not defined`
   or similar, since simply pointing at the on-disk `netstandard.dll`/`mscorlib.dll`
   reference assemblies isn't equivalent to what the SDK normally wires up implicitly.
   `ILRepack.Targets` now builds this helper on demand via `dotnet build` (a fresh
   process, not the in-process `<MSBuild>` task, which was observed to cache a
   pre-restore project evaluation across a `Targets="Restore"` then `Targets="Build"`
   pair within the same build) before invoking it.
6. **`ILRepack`/Mono.Cecil resolves `InputAssemblies` in list order**, and needs each
   assembly's dependencies to already be known to the resolver by the time it reads that
   assembly's custom attributes. `$(TargetPath)` (`Algorand.dll` itself) was listed
   *first*; since it's the one assembly that uses types from every other merged
   dependency, this caused an intermittent-looking (actually deterministic, order-only)
   `Failed to resolve assembly` error depending on which dependency's type a
   custom-attribute argument happened to reference first. Fixed by moving
   `$(TargetPath)` to the end of the `InputAssemblies` list.
7. **`RepackOptions.SearchDirectories` was never populated** (null), causing a
   `NullReferenceException` in `ResolveSearchDirectories()`'s `foreach`. Fixed by always
   passing at least an empty list, and — separately — by actually populating it with
   every directory in `@(ReferencePath)` (fixing a *second*, follow-on failure: ILRepack
   couldn't resolve `Microsoft.Bcl.AsyncInterfaces`, a RestSharp dependency that's
   intentionally *not* merged/internalized, during its reference-fixing pass).
8. **`NuspecBasePath` was set to `$(TargetDir)`**, but `$(TargetDir)` is defined by the
   SDK's common targets (imported at the *bottom* of an SDK-style project), so
   referencing it in a `PropertyGroup` near the *top* of the file silently evaluated to
   empty — the pack step then couldn't find `Algorand.dll` even though it existed
   exactly where expected (`NU5019: File not found: 'Algorand.dll'`, despite the file
   being present — a genuinely confusing failure mode). Fixed by using
   `bin\$(Configuration)\netstandard2.1\` instead, which only depends on
   `$(Configuration)` (a global property, available immediately) and the
   already-pinned `netstandard2.1` target framework for this configuration.
9. **The vendor `ILRepack.Lib.MSBuild.targets` file auto-imports unconditionally into
   *every* build configuration** once the package has ever been restored for this
   project (restore assets aren't per-configuration here), and its own `<ILRepack>`
   target defaults `DoILRepack=true` for any configuration whose name contains
   "Release" — meaning after restoring under `Unity` even once, a subsequent plain
   `Release` build would try to run the *vendor's* broken task too. Fixed by setting
   `<DoILRepack>false</DoILRepack>` unconditionally in the csproj.

## Versioning and CI changes

- `dotnet-algorand-sdk.csproj` now defines a single `$(AlgorandSdkVersion)` property
  used by *both* the regular `Algorand4` package's `<Version>` and the Unity
  configuration's `<Version>`/`<NuspecProperties>`, replacing three previously
  independent, drifted version expressions (`4.7.3.<date>` for the main package,
  `1.0.0.16` for the Unity `<Version>` property, `4.1.2.<date>` in the old
  `nuget pack -Properties version=...` command). The two packages now always ship the
  same version number from the same commit.
- The Unity package is now built via native MSBuild/NuGet.Build.Tasks properties
  (`NuspecFile`/`NuspecBasePath`/`NuspecProperties`) instead of shelling out to the
  classic `nuget.exe pack` CLI command via `PostBuildEvent` — that command required
  `nuget.exe` on `PATH`, which isn't available on the `ubuntu-latest` CI runner (or
  reliably on a fresh dev machine) and, on inspection, was passing a relative
  `Algorand.dll` path that wouldn't have resolved correctly against the actual build
  output directory even where `nuget.exe` was present. This looks like it may never
  have actually produced a working `.nupkg`.
- `.github/workflows/publish-nuget.yml` now also builds the Unity configuration and
  uploads/publishes `Algorand4_Unity`'s `.nupkg` alongside `Algorand4`'s, from the same
  workflow run and commit. The one-time nuget.org Trusted Publishing policy for
  `Algorand4_Unity` (mentioned in the workflow's existing setup comment) still needs to
  be created by a maintainer with nuget.org ownership before the publish step will
  succeed for that package id — this cycle only wires up the *build*, it can't perform
  that one-time manual nuget.org configuration step.

## Should the Unity-specific build be kept, or replaced by the plain `Algorand4` package?

Evaluated per the request to remove the Unity customization if the plain package would
work. Recommendation: **keep it**. `ILRepack.Targets`' `Internalize="true"` exists
specifically so the merged `Newtonsoft.Json`/`MessagePack`/`BouncyCastle.Cryptography`/
etc. types become `internal` inside the single merged `Algorand.dll`, rather than
staying public assemblies a Unity project would need to separately install — Unity
projects very commonly already carry their own (possibly different-versioned) copy of
`Newtonsoft.Json.dll` in particular, and a plain multi-DLL `Algorand4` install would risk
a duplicate-assembly/type conflict that the ILRepack build's internalization is
specifically designed to avoid. This is a real, still-applicable justification for
keeping a dedicated Unity variant, not just historical inertia.

## Recommendations for maintainers

1. Build/install the newly-fixed `Algorand4_Unity` package in an actual Unity Editor
   project and confirm signing/serialization works end-to-end at runtime — this
   environment has no Unity Editor installed, so that final check is still open.
2. Create the nuget.org Trusted Publishing policy for `Algorand4_Unity` (package id)
   before relying on the updated `publish-nuget.yml` to actually push it.
3. Consider filing the `NoRepackRes` MSBuild-task gap upstream against
   `ILRepack.Lib.MSBuild` (or its GitHub source) — other consumers on .NET 9+ will hit
   the same `BinaryFormatter` removal; `BuildTasks/IlRepackMergeTask` is a viable
   permanent workaround but an upstream fix would let other projects avoid needing one.
4. `Microsoft.Build.Utilities.Core` 17.14.8 (a build-time-only dependency of the new
   `BuildTasks/IlRepackMergeTask` project, never shipped in the actual `Algorand4`/
   `Algorand4_Unity` packages) carries an open `NU1903` advisory
   (GHSA-w3q9-fxm7-j8fq). Low real-world risk given it's build-tooling-only and not
   distributed, but worth tracking for a future bump once a compatible newer version
   is confirmed to still load correctly inside the hosting MSBuild version.
