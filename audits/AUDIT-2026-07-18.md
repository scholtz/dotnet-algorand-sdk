## Summary

No regression to the signing/key-material path was found. `Account`/`KeyPair` (key
generation, in-memory key handling) and the Kmd and gossip clients received **no changes**
since the last audit and were not re-derived from scratch this cycle — see Scope covered
below. The ~18 commits since `d9f4315` are dominated by the ARC56 client-generator/ABI
tooling, generated test fixtures, docs/CI workflows, and the Unity build's dependency
versions; none of that touches how private keys are generated, held, or used to sign. Two
new, previously-unrecorded dependency-supply-chain observations were found and added to the
risk register (`RISK-012`, `RISK-013`); one existing open recommendation from the last audit
(`Microsoft.Build.Utilities.Core`'s CVE-2025-55247) is now fixed and closed out
(`RISK-014`, recorded for traceability). Overall verdict: this SDK, as of this commit, does
not put user private keys at materially greater risk than the last audit found.

## Findings

### F1 — Unused Roslyn `using` in a `MessagePack` formatter (dependency itself confirmed legitimate)
- **Severity**: Low
- **Confidence**: Confirmed
- **Status**: New — fixed same day as filed (see below)
- **Location**: `dotnet-algorand-sdk/Algod/Model/Converters/MsgPack/NoDefaultsFormatter.cs:5`
  (`using Microsoft.CodeAnalysis.Host;`).
- **Description**: Nothing in `NoDefaultsFormatter.cs` (`NoDefaultsFormatter<T>`,
  `Deserializer`) references any `Microsoft.CodeAnalysis.*` type — this one `using` is an
  accidental/stray import (e.g. from IDE auto-completion) with no runtime purpose in that
  file. **Correction to this finding's original framing**: the initial pass grepped for
  `Microsoft.CodeAnalysis` usage while excluding the `ClientGenerator/` folder, on the
  assumption those Roslyn-dependent files (`ClientGenerator/Compiler/TealSharpSyntaxWalker.cs`,
  `ClientGenerator/Compiler/CompilerStates/*.cs`) belong only to the separate
  `ClientGenerator/AlgoStudio.csproj` project. Attempting the fix (removing the three
  `Microsoft.CodeAnalysis.CSharp`/`.CSharp.Workspaces`/`.Workspaces.Common`
  `PackageReference`s from `dotnet-algorand-sdk.csproj`) and rebuilding
  (`dotnet build dotnet-algorand-sdk.sln --configuration Release`) proved that assumption
  wrong: those files have no `<Compile Remove>` exclusion in `dotnet-algorand-sdk.csproj`,
  so SDK-style default globbing compiles them into the main project too (819 `CS0246`
  errors for Roslyn syntax types — `ISymbol`, `NamespaceDeclarationSyntax`,
  `BlockSyntax`, etc. — confirmed this on removal). **The three
  `Microsoft.CodeAnalysis.*` package references are genuinely required by the main
  `Algorand4` package and were restored; only the one stray `using` was removed.**
- **Failure scenario**: N/A for the dependency itself (needed, not bloat). The only actual
  issue was the single unused `using` statement — cosmetic, no behavior or supply-chain
  impact once corrected.
- **Fix applied**: Removed `using Microsoft.CodeAnalysis.Host;` from
  `NoDefaultsFormatter.cs`. Verified: `dotnet restore` and
  `dotnet build dotnet-algorand-sdk.sln --configuration Release --no-restore` both succeed
  with 0 warnings / 0 errors, including the `Algorand4` package build and pack step.
- **Recommendation**: None further — resolved. `RISK-013` (originally filed as a dependency-
  bloat risk) is corrected to reflect that the Roslyn dependencies are legitimate; see risk
  register changes below.

### F2 — Canonical transaction encoding (the signing path) depends on an unmaintained, single-maintainer bridge package
- **Severity**: Low
- **Confidence**: Confirmed
- **Status**: New
- **Location**: `dotnet-algorand-sdk/Utils/Encoder.cs:8` (`using Newtonsoft.Msgpack;`),
  used by `EncodeToMsgPackOrdered`/`EncodeToMsgPackNoOrder` (lines ~30-93, the
  `MessagePackWriter`/`JsonSerializer.Serialize(writer, o)` calls that resolve to
  `Newtonsoft.Msgpack`, not the `MessagePack` CSharp library, since this file has no
  `using MessagePack;`) and by the decode path (`DecodeFromMsgPack`, `MessagePackReader`).
  `dotnet-algorand-sdk.csproj:192`: `<PackageReference Include="Newtonsoft.Msgpack" Version="0.1.11" />`.
- **Description**: Traced the exact code path: every canonical msgpack encoding of a
  transaction prior to signing (`EncodeToMsgPackOrdered`) goes through
  `Newtonsoft.Msgpack`'s `MessagePackWriter`, not just the decode/display path. Checked the
  package's NuGet listing: single maintainer (`darkl`), 12 releases total, last published
  **2019-08-31** (over 6 years before this audit date), and `0.1.11` is still the latest
  version on nuget.org — there is no newer release to move to. The package also has no
  strong-named build (see the `CS8002`-suppression comment added in `dotnet-algorand-sdk.csproj`
  by commit `b912e53`, which is itself an accurate description of this gap). This is a
  distinct, previously-unrecorded observation from `RISK-002` (which covered
  `BouncyCastle.Cryptography`, now on the official pinned package) — this package sits in
  the same signing-critical code path but was not called out in any prior audit.
  For context, `AUDIT-2026-07-16.md`/`RISK-002` already established the general principle
  that crypto/signing-relevant dependencies need standard CVE tracking; an abandoned
  single-maintainer package is the same category of concern even though no specific CVE is
  known against it today.
- **Failure scenario**: If a vulnerability were ever found in `Newtonsoft.Msgpack`'s
  encode/decode logic (e.g. a malformed-input parsing bug reachable via
  `DecodeFromMsgPack`/`DecodeFromJson` on attacker-influenced node responses), there is no
  active maintainer to patch it and no newer version to upgrade to — a downstream consumer
  would be stuck either accepting the risk or having to fork/patch the dependency
  themselves. This is a supply-chain continuity risk, not a currently-known live
  vulnerability.
- **Recommendation**: Track this as a standing supply-chain risk (added as `RISK-012`).
  No immediate action is required absent a known CVE, but a maintainer should periodically
  re-check whether `Newtonsoft.Msgpack` has resumed releases, whether a maintained fork
  exists, or whether the encode/decode logic it provides could be replaced with
  `MessagePack`-CSharp's own APIs (already a direct dependency) to remove the dependency
  entirely — a non-trivial refactor given `Newtonsoft.Msgpack`'s `JsonSerializer`+
  `ContractResolver`-based approach is what gives the ordered-field-encoding behavior the
  canonical-encoding tests depend on, so this is not a "trivial swap."

- **Status**: Resolved since last audit (commit `b912e53`)
- **Location**: `BuildTasks/IlRepackMergeTask/IlRepackMergeTask.csproj`

## Positive note (no finding, verified correct)

`Utils/Encoder.cs`'s new `MayBeMisresolvedKeyreg<T>` helper (added in commit `e97adbe`) was
read in full and traced: `KeyRegisterOfflineTransaction`'s fields are a strict subset of
`KeyRegisterOnlineTransaction`'s, so MessagePack-CSharp's contractless `[Union]` resolver
can silently misresolve an online key-registration transaction as an offline one when
decoding raw wire bytes (no discriminator byte exists in Algorand's canonical field-map
encoding). The fix re-decodes via the `Newtonsoft.Msgpack`+`JsonSubtypes` path (which does
honor the `KnownSubTypeWithProperty` discriminator already declared on
`KeyRegistrationTransaction`) whenever the fast path resolves to `Offline`, since a
genuinely-offline transaction decodes identically either way and the extra decode pass is
cheap. This is a decode/type-fidelity correctness fix — it does not affect signing
(encoding is unaffected) and closes a real, if narrow, misclassification bug rather than
introducing risk.
