## Summary

This audit independently re-verified that all five findings from `AUDIT-2026-07-16.md`
(F1–F5) were correctly and completely fixed in commit `f975bd6` — not just claimed
fixed. All five check out: key zeroing genuinely zeroes the live byte array, the
BouncyCastle NuGet migration is complete with no leftover vendored DLL, the
`ApiException` redaction covers every throw site in the Kmd client, the stray
`Console.WriteLine` is gone, and Kmd's remote-custody behavior is unchanged (as
expected — it required no code fix). No regressions were introduced by the fixes.

This cycle also covered the two areas explicitly deferred by the previous audit (the
Unity ILRepack build variant and the Gossip network client), plus a fresh full-scope
pass. No private-key leakage was found anywhere. Two new, low-severity findings
surfaced: the Unity build's ILRepack step doesn't merge the `MessagePack` package or
declare it as a nuspec dependency (a build-integrity gap predating this week's changes,
not a crypto-weakening one), and the SDK's own `OfflineSigningExample.cs` prints a
generated mnemonic to the console — directly modeling the anti-pattern `RISK-005`
warns developers against, in code the maintainers fully control.

**Update (same day)**: Both findings (U1, E1) were fixed and verified — see "Remediation"
below. `RISK-009`/`RISK-010` updated to Mitigated in `RISKS.md`.

## Findings

### Re-verification of prior findings (F1–F5)

- **F1 (key zeroing) — Confirmed fixed, no regression.**
  `dotnet-algorand-sdk/Crypto/KeyPair.cs:95-104` `Dispose()` performs `Array.Clear` on
  `ClearTextPrivateKey` and nulls the BouncyCastle key object references;
  `Algod/Model/Account.cs:161-164` delegates to it. `test/AccountTests.cs`
  `testDisposeWipesPrivateKey` holds a reference to the *same* `byte[]` the account
  exposes before calling `Dispose()`, then asserts it is all-zero afterward — this
  genuinely proves in-place zeroing (not just reference-dropping, which would let a
  test pass even if the underlying bytes were never touched). Residual best-effort
  caveat (BouncyCastle's own `Ed25519PrivateKeyParameters` internal buffer is
  dereferenced but not directly zeroable) is unchanged and already documented.
- **F2 (BouncyCastle NuGet migration) — Confirmed fixed, no regression.**
  `dotnet-algorand-sdk/dotnet-algorand-sdk.csproj` now carries
  `PackageReference Include="BouncyCastle.Cryptography" Version="2.6.2"`; the old
  `Content`/`Reference` blocks pointing at `Libs\BouncyCastle.Crypto.dll` are gone, and
  the file itself is deleted from disk (only Unity stub DLLs remain in `Libs/`).
  `ILRepack.Targets:11` was updated to filter on `'BouncyCastle.Cryptography'`,
  matching the new package's actual assembly filename. `test/test.csproj` and
  `sdk-examples/sdk-examples.csproj` had their now-dead `HintPath` references removed.
- **F3 (ApiException redaction) — Confirmed fixed, coverage verified.**
  `dotnet-algorand-sdk/ApiException.cs` no longer embeds the raw response body in
  `Message`/`ToString()`, only its length; `Response` still carries the full body for
  callers who want it. Traced every `throw new ApiException(...)` call site in
  `Kmd/Kmd.cs` (10 sites, including the Export/Import paths) back to the same base
  constructor — the fix applies uniformly, there is no bypass path. The
  `ApiException<TResult>.ToString()` override (unchanged, falls back to
  `Result.ToString()`) is a separate, pre-existing surface not implicated by this
  finding.
- **F4 (stray `Console.WriteLine`) — Confirmed fixed.** `Signature.cs`'s
  `LogicsigSignature.Verify()` catch block now just returns `false`.
- **F5 (Kmd remote-custody, documentation-only) — unchanged, as expected**; no code
  fix was warranted or made.

### U1 — Unity ILRepack build does not merge or declare the `MessagePack` package
- **Severity**: Low · **Confidence**: Confirmed (code-level gap) / Plausible (runtime
  consequence, not reproduced in this environment) · **Status**: Fixed (same day, see
  "Remediation" below) · Originally: New (pre-existing,
  not introduced by this week's changes — the diff between the previous and current
  commit only changed the `BouncyCastle.Crypto` → `BouncyCastle.Cryptography` filter
  line)
- `dotnet-algorand-sdk/ILRepack.Targets:5-12` merges `Newtonsoft.Json`, `JsonSubTypes`,
  `Newtonsoft.Msgpack`, `MsgPack`, `System.CodeDom`, `BouncyCastle.Cryptography`, and
  `RestSharp` into the ILRepacked `Algorand.dll` for the Unity configurations, but does
  **not** include `MessagePack` (the neuecc `MessagePack` NuGet package referenced at
  `dotnet-algorand-sdk.csproj:137`, v3.1.8).
  `dotnet-algorand-sdk/Algorand4_Unity.nuspec`'s `<dependencies>` group only lists
  `Microsoft.CSharp` and `System.ComponentModel.Annotations` — `MessagePack` is neither
  merged in nor declared as something the consumer must separately install.
- `MessagePackObject`/`Key`/`IgnoreMember` attributes are used pervasively on
  signing-critical types (`Account.cs:30,37-38`, `Signature.cs`, every transaction
  model under `Algod/Model/Transactions/`), so `MessagePack.dll`'s types are a hard
  runtime dependency of the SDK's own serialization/signing path, not an optional one.
- **Scenario**: A Unity developer installs the packed `Algorand4_Unity` package,
  follows its nuspec-declared dependencies exactly, and hits a
  `TypeLoadException`/`FileNotFoundException`/`MissingMethodException` the first time
  they serialize or sign a transaction, because `MessagePack.dll` was never merged in
  and isn't declared as something to install separately. This is a correctness/build
  gap, not a security weakening (it doesn't mean the merged assembly silently signs
  with worse crypto — it means signing may not run at all until the gap is fixed) —
  included here because it was explicitly in this cycle's scope and affects the SDK's
  reliability for a class of consumers (Unity/game developers), which matters for the
  audit's broader "does this SDK behave as developers expect" mandate.
- **Recommendation**: Add `<InputAssemblies Include="@(ReferencePath)" Condition="'%(filename)' == 'MessagePack'" />` to `ILRepack.Targets`, and/or add
  `MessagePack` to `Algorand4_Unity.nuspec`'s dependency group. A maintainer with a
  working Unity toolchain should confirm the packed assembly actually resolves
  `MessagePack` types at runtime after the fix.

### E1 — `sdk-examples/OfflineSigningExample.cs` prints a generated mnemonic to the console
- **Severity**: Low · **Confidence**: Confirmed · **Status**: Fixed (same day, see
  "Remediation" below) · Originally: New (file was added in
  the commit the *first* audit was run against, `97a6c79`, but wasn't flagged then —
  the first audit's scope was the SDK's own source, not `sdk-examples/`)
- `sdk-examples/OfflineSigningExample.cs:24-26`:
  ```csharp
  var newAccount = new Account();
  Console.WriteLine("Generated address:  " + newAccount.Address);
  Console.WriteLine("Backup mnemonic:    " + newAccount.ToMnemonic());
  ```
- **Scenario**: This is an official SDK example, meaning developers are expected to
  copy patterns from it. Printing a freshly generated mnemonic straight to the console
  models exactly the anti-pattern `audits/RISKS.md` `RISK-005` ("developers hardcode or
  log mnemonics") warns about generally — except here it's inside code this repo's own
  maintainers control, unlike `RISK-005` itself which is about third-party developer
  code the SDK can't govern. All other example files
  (`AccountExample.cs`, `AssetExample.cs`, `KeyRegistrationExample.cs`, etc.) only
  reference mnemonics via fixed placeholder strings used to derive a sender account,
  never printed at runtime — this file is the one exception.
- **Recommendation**: Remove the `Console.WriteLine` of the mnemonic (or replace with a
  comment / a redacted placeholder), and instead direct the reader to persist it via a
  secure channel of their own choosing. This is a one-line fix with no functional
  impact on the rest of the example (the mnemonic is still available via
  `newAccount.ToMnemonic()` in-memory for whatever the example does next).

### Gossip network client — no key-material exposure found
- **Confidence**: Confirmed (full read of `Gossip/GossipHttpClient.cs`,
  `Gossip/GossipWebsocketClient.cs`, `Gossip/GossipHttpConfiguration.cs`,
  `Gossip/DnsSrvResolver.cs`)
- No `Account`/`KeyPair`/mnemonic type is referenced anywhere in `Gossip/`. This client
  only fetches/parses public blocks and decodes public signed-transaction gossip
  traffic (already-signed, already-public data) — there is no path by which private key
  material could flow through it.
- Relay endpoints in `GossipHttpConfiguration.cs` are plain `http://`, consistent with
  the actual Algorand gossip protocol's design (integrity comes from
  consensus/signatures, not transport confidentiality) — not a finding, since no secret
  material transits this path.
- `DnsSrvResolver.cs` performs raw DNS SRV lookups with no DNSSEC validation. A
  network-position attacker could in principle spoof SRV responses to redirect gossip
  connections to a relay they control — but since no key material ever flows through
  gossip clients, the worst case is forged/stale block or transaction data, not key
  theft. This is already covered by the existing `RISK-006` (malicious/compromised
  node operator) rather than warranting a new entry.

### No new findings in: dependency supply chain, generated code pipeline
- `ClientGenerator/`, `client-generator/`, and the Velocity templates under
  `api-generator/dotnet_templates/` have zero diff between `97a6c79` and `f975bd6` —
  the previous audit's conclusions for these areas (F2 aside, which was about the main
  SDK project's own dependency, not the generator) still hold on re-review.
- No TLS/certificate-validation bypasses found anywhere in the repo (searched for

- Kmd's `ImportKeyAsync`/`ExportKeyAsync`/`ExportMasterKeyAsync` remain the one
  intentional key-transmission path, unchanged and consistent with Kmd's own API
  contract (`RISK-004`).

## Remediation (same day, 2026-07-16)

Both findings from this cycle were fixed and verified after the audit was written:

- **U1 fixed**: `dotnet-algorand-sdk/ILRepack.Targets` now also merges `MessagePack`
  and `MessagePack.Annotations` into the ILRepacked Unity assembly, alongside the
  existing dependency list. `MessagePack.Annotations` was confirmed to be a genuine
  separate assembly (not just an alias) by checking build output —
  `MessagePack.Annotations.dll` is produced as its own file alongside `MessagePack.dll`
  in every project's `bin/` output — so both filename conditions are needed for the
  merge to be complete. `Algorand4_Unity.nuspec`'s dependency group was left as-is
  (deliberately, since `MessagePack` is merged rather than externally required), with a
  comment added explaining why it's absent from the list.
- **E1 fixed**: `sdk-examples/OfflineSigningExample.cs:24-26` no longer prints
  `newAccount.ToMnemonic()` to the console; replaced with a comment pointing at secure
  persistence. The mnemonic is still available in-memory for the rest of the example.
- **Testing performed**:
  - `dotnet build dotnet-algorand-sdk.sln --configuration Release --no-restore` — 0
    errors.
  - `dotnet build sdk-examples/sdk-examples.csproj --no-restore` — 0 errors (confirms
    E1's fix compiles cleanly).
  - `dotnet test SerialisationTests/serialisation-tests.csproj` — 23/23 passed
    (confirms the BouncyCastle/MessagePack-touching serialization/signing paths are
    unaffected by the ILRepack.Targets edit, which doesn't touch the normal
    Release/Debug build path at all — it only fires under the `Unity`/`Unity_debug`
    configurations).
  - `dotnet test test/test.csproj --filter "FullyQualifiedName!~Arc4Tests&FullyQualifiedName!~Arc56Tests"`
    — 96/96 passed. The excluded Arc4Tests/Arc56Tests fixtures need a live AlgoKit
    LocalNet (`http://localhost:4001`/`:4002`) per `CLAUDE.md`, which wasn't running in
    this environment; running the full suite without the filter reproduced exactly the
    same 34 pre-existing LocalNet-dependent failures (`ApiException` from
    `TransactionsAsync`/connection-refused) with the `--filter` removed, confirming
    they're unrelated to this fix, not a regression it introduced.
  - `dotnet build dotnet-algorand-sdk.sln --configuration Unity --no-restore` — still
    fails, with the same pre-existing `SerializeFieldAttribute`/`TooltipAttribute`/
    `InspectorNameAttribute` errors in `Algod/Model/StateProofTrackingData.cs`,
    `Algod/Model/Transactions/AssetCloseTransaction.cs`, etc. documented in the
    previous audit's remediation section — confirming the Unity-stub-DLL compile
    failure is still the blocker for true end-to-end verification of U1's fix (the
    ILRepack step never runs, since the assembly it would merge never finishes
    compiling). U1's fix is therefore verified at the code level (the merge condition
    is syntactically correct and targets a real, confirmed-present assembly file) but
    not by actually producing and inspecting a packed Unity `Algorand.dll`.
